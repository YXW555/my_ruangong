## 敏捷开发实践

### 1. 看板管理（Kanban）

使用 GitHub Projects 或 Trello 管理任务：

**看板列**：
- 📋 **待办（Backlog）**：所有任务
- 🔄 **进行中（In Progress）**：正在做的任务
- 🧪 **测试中（Testing）**：等待测试
- ✅ **已完成（Done）**：已完成

### 2. 用户故事（User Story）

每个功能用用户故事描述：

**格式**：
```
作为 [用户角色]
我想要 [功能]
以便 [价值]
```

**示例**：
- 作为学生，我想要快速搜索商品，以便找到想要的物品
- 作为卖家，我想要批量上架商品，以便提高效率
- 作为管理员，我想要查看数据统计，以便了解平台运营情况

### 3. 任务优先级

**P0（必须）**：核心功能，没有这个系统无法使用
- 用户注册/登录
- 商品发布/浏览
- 订单创建/查看

**P1（重要）**：重要功能，影响用户体验
- 商品搜索
- 图片上传
- 订单管理

**P2（可选）**：增强功能，有更好
- 数据统计
- 商品置顶
- 高级筛选

### 4. 快速迭代

**每个迭代（1天）**：
1. **计划会议**（30分钟）：分配任务，明确目标
2. **开发**（6-7小时）：实现功能
3. **测试**（1小时）：测试功能
4. **回顾会议**（30分钟）：总结问题，调整计划

### 5. 持续集成

**每天至少提交一次代码**：
- 小功能完成就提交
- 不要等到一天结束
- 使用有意义的提交信息

**提交格式**：
```
feat: 添加商品搜索功能
fix: 修复图片加载问题
refactor: 优化代码结构
```

## 风险控制

### 风险 1：功能太多，做不完

**应对**：
- 优先做 P0 功能
- 如果做不完，先做核心流程
- 其他功能可以简化或去掉

### 风险 2：前后端联调问题

**应对**：
- 提前定义接口规范
- 使用 Mock 数据先开发前端
- 每天至少联调一次

### 风险 3：部署问题

**应对**：
- 第 3 天就开始测试部署
- 不要等到最后一天
- 准备备用方案

## 敏捷开发核心价值

1. **快速迭代**：每天一个版本，4天完成4个迭代
2. **持续交付**：每天都有可用的版本
3. **用户价值优先**：先做最重要的功能
4. **快速反馈**：每天站会，及时调整
5. **拥抱变化**：根据实际情况灵活调整计划
